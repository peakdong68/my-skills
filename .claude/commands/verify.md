调用 my-agent-skills:workflow-verification 技能。

该命令是工作流收口步骤，在 `/review` 之后、`/ship` 之前执行。不信任 todo.md 的当前 `[x]` 状态——从 git log 完全反推实际完成情况。分四个阶段：

## 阶段 1: 采集信号
- 读取 `docs/plans/<feature>/plan.md` — 计划与检查点定义（只读）
- 读取 `docs/plans/<feature>/todo.md` — 任务骨架（忽略现有 `[ ]/[x]` 状态）
- ★ 涉及 git 操作前，先调用 `my-agent-skills:git-workflow-and-versioning` 获取提交格式规范与历史查询模式
- 读取 `git log`（从计划创建至今，按 git-workflow-and-versioning 的提交格式解析）— 唯一可信的执行事实来源
- 读取 `/review` 输出中的偏差记录表格
- 采集当前代码状态（`git status`、`git diff` 未暂存变更、构建/测试结果）

## 阶段 2: 反推任务完成状态
- 用提交消息关键词 + 变更文件路径，逐提交映射到计划任务
- 任务状态标记：`[x]` 完成 / `[~]` 部分 / `[ ]` 未完成 / `[+]` 新增 / `[-]` 废弃
- 逐检查点验证条件是否满足

## 阶段 3: 产物同步
- ★ 完全重写 `todo.md` — 将所有 `[ ]` 替换为实际判定标记
- 更新 `plan.md` — 检查点结果 + 追加 `## 执行偏差` 章节

## 阶段 4: 验证报告
- 产出结构化验证报告：执行摘要、检查点结果、任务同步详情、开放项、就绪判定

如果发现致命阻塞项（关键检查点未通过），在报告中明确标注并阻止进入 `/ship`。
