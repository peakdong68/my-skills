# 技能结构说明

本文档描述了 agent-skills 技能文件的结构与格式。在贡献新技能或理解现有技能时，请将其作为指南。

## 文件位置

每个技能都位于 `skills/` 目录下的独立文件夹中：

```
skills/
  skill-name/
    SKILL.md           # 必填：技能定义
    supporting-file.md # 可选：按需加载的参考资料
```

## SKILL.md 格式

### 前置元数据（必填）

```yaml
---
name: 全小写短横线分隔的技能名
description: 引导 Agent 完成 [任务/工作流]。在 [具体触发条件] 时使用。
---
```

**规则：**

- `name`：全小写，短横线分隔。必须与目录名称一致。
- `description`：以第三人称开头说明该技能的功能，随后包含一个或多个明确的“使用条件”触发条件。需同时涵盖*做什么*和*何时做*。最多 1024 个字符。

**为何重要：** Agent 通过阅读描述来发现技能。描述会被注入到系统提示词（system prompt）中，因此必须明确告知 Agent 该技能提供什么功能以及何时激活。请勿在此总结工作流——如果描述中包含流程步骤，Agent 可能会直接遵循摘要，而不会去阅读完整的技能文件。

### 标准章节（推荐结构）

```markdown
# 技能标题

## 概述

用一两句话说明该技能的功能及其重要性。

## 何时使用

- 触发条件的要点列表（症状、任务类型）
- 何时不应使用（排除情况）

## [核心流程 / 工作流 / 步骤]

主要工作流，拆分为编号步骤或阶段。
在有帮助的地方包含代码示例。
在存在决策点的地方使用流程图（ASCII）。

## [特定技巧 / 模式]

针对具体场景的详细指导。
代码示例、模板、配置。

## 常见合理化借口

| 借口                     | 实际情况         |
| ------------------------ | ---------------- |
| Agent 用来跳过步骤的借口 | 为何该借口不成立 |

## 危险信号

- 表明技能正在被违反的行为模式
- 审查期间需要留意的事项

## 验证

完成技能流程后，请确认：

- [ ] 退出标准清单
- [ ] 证据要求
```

## 章节作用说明

### 概述

技能的“电梯演讲”（简短推介）。应回答：该技能做什么？为什么 Agent 需要遵循它？

### 何时使用

帮助 Agent 和人类判断该技能是否适用于当前任务。需同时包含正向触发条件（“当 X 时使用”）和负向排除条件（“不适用于 Y”）。

### 核心流程

技能的核心。这是 Agent 需逐步遵循的工作流。必须具体且可执行——避免模糊的建议。

**正确示例：** “运行 `npm test` 并验证所有测试是否通过”
**错误示例：** “确保测试能跑通”

### 常见合理化借口

精心编写的技能中最具特色的部分。这些是 Agent 用来跳过重要步骤的借口，并附带反驳理由。它们能防止 Agent 通过自我合理化来逃避遵循流程。

回想一下 Agent 每次说“我稍后再加测试”或“这很简单，可以跳过规范”的情况——将这些借口连同事实依据的反驳论点写在这里。

### 危险信号

可观察到的技能被违反的迹象。在代码审查和自我监控时非常有用。

### 验证

退出标准。Agent 用于确认技能流程已完成的检查清单。每个复选框都应能通过证据进行验证（测试输出、构建结果、截图等）。

## 辅助文件

仅在以下情况下创建辅助文件：

- 参考资料超过 100 行（保持主 SKILL.md 内容聚焦）
- 需要代码工具或脚本
- 清单足够长，值得拆分为独立文件

当模式或原则不足 50 行时，请直接内联写在主文件中。

## 编写原则

1. **重流程，轻知识。** 技能是工作流，而非参考文档。写步骤，而非罗列事实。
2. **重具体，轻笼统。** “运行 `npm test`” 优于 “验证测试”。
3. **重证据，轻假设。** 每个验证复选框都必须有证明依据。
4. **防自我合理化。** 任何可能被跳过的步骤，都必须在借口表中提供反驳论点。
5. **渐进式披露。** 主 SKILL.md 是入口。辅助文件仅在需要时加载。
6. **注重 Token 效率。** 每个章节都必须证明其存在的必要性。如果删除它不会改变 Agent 的行为，就删掉它。

## 命名规范

- 技能目录：`全小写短横线分隔`
- 技能文件：`SKILL.md`（始终大写）
- 辅助文件：`全小写短横线分隔.md`
- 参考资料：存放在项目根目录的 `references/` 中，而非技能目录内

## 跨技能引用

通过名称引用其他技能：

```markdown
编写测试时，请遵循 `test-driven-development` 技能。
如果构建失败，请使用 `debugging-and-error-recovery` 技能。
```

请勿在不同技能间重复内容——改为使用引用和链接。
