# spec-review 技能设计

## 问题陈述

**我们如何能**在 spec-driven-development 产出规范之后、planning-and-task-breakdown 开始之前，插入一个由**独立代理实例**执行的 Spec Review 步骤，对格式、一致性和边界条件进行三层校验，从源头阻断不完整规范向下游扩散？

## 推荐方向：独立技能 + 双触发（自动 + 手动）

创建独立 `spec-review` 技能和 `/spec-review` 命令。提供两种触发路径：

1. **自动触发**：`spec-driven-development` 步骤 9（用户批准后、过渡到 plan 前），建议运行 spec-review
2. **手动触发**：用户随时通过 `/spec-review` 命令调用

审查必须由**独立的子代理实例**执行——不能是写 spec 的同一个代理。软门控：产出结构化报告，由用户决定是否继续进入 plan。

## 三层校验体系

| 层级 | 名称 | 检查内容 | 严重级别 |
|------|------|---------|---------|
| L1 | 结构完整性 | 6 核心领域是否全部存在、文档路径是否正确、格式标记完整 | 严重 |
| L2 | 需求一致性 | Spec 行为是否覆盖了原始需求中所有功能点、是否无内部矛盾 | 严重 |
| L3 | 边界与鲁棒性 | 异常场景、边界输入、错误处理、空/加载状态、安全考量 | 重要 |

每项检查结果分类为：**严重**（阻塞级——强烈建议修复后再 plan）、**重要**（应修复但可酌情继续）、**建议**（可选改进）。

## 审查流程（技能内部步骤）

```
1. 定位 Spec 文档 → 2. 提取原始需求 → 3. L1 结构完整性 → 4. L2 需求一致性 →
5. L3 边界与鲁棒性 → 6. 产出审查报告 → 7. 呈现报告给用户
```

### 步骤详情

**1. 定位 Spec 文档**
- 从 `docs/specs/` 下找到最新的 spec 文档（按日期前缀）
- 如果用户指定了文件路径，使用指定路径
- 如果没有找到，报告错误并退出

**2. 提取原始需求**
- 如果是从 idea-refine 产出，检查 `docs/ideas/` 下关联的想法文档
- 如果需求来自对话，从 spec 文档的"目标"章节提取原始需求清单
- 建立"需求 → spec 覆盖"的映射基线

**3. L1 结构完整性检查**
- 6 核心领域（目标/命令/结构/风格/测试/边界）是否全部存在
- 技术栈、成功标准、待解决问题、不做事是否均有内容
- 文档路径是否在 `docs/specs/` 下
- 是否有空章节或占位符内容

**4. L2 需求一致性检查**
- spec 中描述的行为是否覆盖了需求想法中的每个功能点
- 成功标准是否可测量（非"让它更好"之类的模糊描述）
- 是否存在内部矛盾（如技术栈说 PostgreSQL，命令却是 MongoDB 相关）
- 边界定义是否与功能需求一致

**5. L3 边界与鲁棒性检查**
- 是否考虑了异常场景（网络失败、API 错误、超时）
- 是否考虑了边界输入（空输入、超长输入、特殊字符）
- 是否定义了错误处理策略
- 是否有空状态、加载状态、错误状态的 UI 处理
- 是否有安全相关考量（用户输入验证、认证/授权）

**6. 产出审查报告**（见下方模板）

**7. 呈现报告给用户**
- 报告以表格形式呈现每项检查结果
- 每个问题附带具体位置（spec 章节）和修复建议
- 汇总结论：✓ 可以开始计划 / ⚠ 有建议但可继续 / ✗ 建议修复后再计划

## 审查报告模板

```markdown
# Spec Review: [spec 文件名]

**审查时间：** [时间戳]
**审查实例：** [独立子代理标识]

## 总体评估

**结果：** ✓ 可以开始计划 / ⚠ 有建议但可继续 / ✗ 建议修复后再计划

## L1 结构完整性

| # | 检查项 | 结果 | 位置 | 说明 |
|---|--------|------|------|------|
| 1 | 目标章节完整 | ✓ | §目标 | — |
| 2 | 命令可执行 | ✓ | §命令 | — |
| ... | ... | ... | ... | ... |

## L2 需求一致性

| # | 需求项 | 覆盖情况 | spec 章节 | 说明 |
|---|--------|---------|----------|------|
| 1 | 用户登录功能 | ✓ 已覆盖 | §目标 | — |
| 2 | 数据导出 CSV | ✗ 未覆盖 | — | 需求提到但 spec 未涉及 |
| ... | ... | ... | ... | ... |

## L3 边界与鲁棒性

| # | 场景 | 结果 | 说明 |
|---|------|------|------|
| 1 | 网络异常处理 | ⚠ 未定义 | 未说明 API 调用失败时的行为 |
| 2 | 空数据状态 | ✗ 缺失 | 未定义列表为空时的 UI 表现 |
| ... | ... | ... | ... |

## 修复建议

1. [严重] 需求"数据导出 CSV"在 spec 中未覆盖 → 在 §目标 中添加对应验收标准
2. [重要] 未定义网络异常处理策略 → 在 §边界 中添加错误处理策略
3. [建议] 成功标准"性能良好"不够可测量 → 改为具体指标（如 LCP < 2.5s）
```

## 独立实例机制

在 spec-review 技能的执行说明中明确要求：

> 审查必须由独立子代理执行。当本技能被触发时，**必须**使用 Agent 工具派发一个独立子代理来执行审查流程。不得在当前会话中直接执行审查——这会违反"不能自己检查自己"的原则。

建议配置：
- 子代理类型：`general-purpose`
- 模型：Haiku（快速扫描）或 Sonnet（深度审查）
- 输入：spec 文件路径 + 原始需求上下文

## 需验证的关键假设

- [ ] Claude Code 的 Agent 子代理机制可以在 SKILL.md 中被可靠触发——验证：实际调用 `/spec-review` 并确认子代理被派发
- [ ] 独立子代理能否正确访问 `docs/specs/` 下的 spec 文件——验证：子代理执行 Read 操作确认
- [ ] 审查报告格式是否对用户可读——验证：产出一次完整报告并人工评估
- [ ] 软门控是否真正被用户使用——验证：观察用户是"看了报告就跳过"还是"根据报告修复后重新审查"

## MVP 范围

**包含：**
- `skills/spec-review/SKILL.md`（~200 行，三层校验流程）
- `.claude/commands/spec-review.md`（~15 行命令入口）
- 对 `spec-driven-development` 步骤 9 的微量修改（+1 行建议引用）
- 对 `planning-and-task-breakdown` 步骤 1 的微量修改（+1 行前置检查引用）

**排除：**
- `agents/spec-reviewer.md`（MVP 后优化）
- 自动化门控（MVP 用软建议，不阻塞流程）
- 与 workflow-verification 的深度集成
- CI/CD 集成

## 不做（及原因）

- **不创建 spec-reviewer 角色** —— MVP 用 general-purpose 子代理跑审查流程。角色是优化项，不是 MVP 必须
- **不做硬门控** —— 用户明确选择软建议。硬门控会过度复杂化第一个版本
- **不修改 `/spec` 命令文件** —— 命令文件只需引用技能名，技能内容变更对其透明。额外的自动触发逻辑写在 spec-driven-development 技能内部
- **不在审查报告中给"通过/不通过"判决** —— 只给分类建议，决策权归用户。这与"软建议"原则一致

## 需求来源策略

审查子代理获取"原始需求"的双路径：

1. **主路径——对话上下文传递：** 派发审查子代理时，将主会话中用户表达的原始需求、澄清问题和确认结论作为 `prompt` 的一部分传递给子代理。子代理以此为基线，对照 spec 文档执行 L2 一致性检查
2. **补充路径——idea-refine 文档：** 如果需求先经过了 idea-refine 流程，检查 `docs/ideas/` 下关联的想法文档，提取其中的需求清单作为额外参考
3. **兜底路径——spec 目标章节反推：** 如果对话上下文不可用（如手动调用 `/spec-review`），从 spec 的"目标"章节反推需求清单，但这标记为"低置信度基线"

## 大型 Spec 分批审查

对于涵盖多个独立子系统的 spec，采用**分批审查**：

1. 审查子代理首先执行"范围评估"——检查 spec 是否描述了多个独立子系统
2. 如果识别到 2+ 个独立子系统，按子系统拆分为单独的审查单元
3. 每个子系统分别执行 L1/L2/L3 全套检查，产出独立的审查报告章节
4. 汇总结论覆盖所有批次

触发条件：spec 中明确声明了多个子系统（如"这个项目包含：聊天系统、文件存储、计费平台"），或 spec-driven-development 步骤 1 中已标记为需要拆分的项目
