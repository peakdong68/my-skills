# /verify — 工作流收口验证与产物同步

## 问题陈述

我们如何能在 `/review` 之后、`/ship` 之前，系统化地验收 todo.md 任务清单、检查 plan.md 验证点、补全遗漏标记，并在原始计划因执行中演变（审查修复、新增任务、ship 修复等）后，将 todo.md 和 plan.md 重写为最终真实状态，同时产出可追溯的验证报告？

## 根因：todo.md 从创建那一刻就死了

核查发现，**整个命令链中没有任何一个环节真正更新 todo.md**：

```
/planning → 产生 todo.md（全是 [ ]）
/build → 提交代码，todo.md 原封不动 ← 步骤 8 "标记任务为完成" 是死文字
/test  → 运行测试，不碰 todo.md
/review → 审查代码，不碰 todo.md
/ship → 检查质量，不碰 todo.md
```

`/build` 的步骤 8 写了「标记任务为完成（todo任务列表(task list)产物文档）」，但它调用的下层技能（`incremental-implementation`、`test-driven-development`）都没有编辑 todo.md 的指令。实际操作中 agent 只做实现→测试→提交，从未触及计划产物。

这意味着 `/verify` **不能依赖任何中间步骤的标记**。它必须从唯一可信的事实来源——git 提交历史和代码实际状态——完全反推任务完成情况。

## 推荐方向

在现有命令链 `/spec → /plan → /build → /test → /review → /ship` 中，**在 `/review` 之后插入新命令 `/verify`**，作为唯一且完整的收口验证步骤。它不信任 todo.md 的当前状态，而是从原始信号重建真实情况：

- **git log**（唯一可信的执行记录）→ 反推哪些任务实际完成了
- **plan.md**（原始计划 + 检查点定义）→ 对比计划的「应有」vs 实际的「已有」
- **当前代码状态**（构建 + 测试结果）→ 验证检查点条件
- `/review` 输出中的偏差记录 → 补充 git log 无法体现的信息（bug 修复、返工）

最后以重建的真实状态**完全重写** todo.md 和 plan.md，并产出验证报告。

同时修正 `git-workflow-and-versioning` 技能的触发条件——从无效的「任何代码更改时」改为精确的内部触发：在 `/build` 提交边界和 `/verify` 历史分析时调用。

## 工作流变更一览

```
现状 — todo.md 全程无人触碰：
/spec → /plan → /build → /test → /review → /ship
  │       │        │         │        │         │
  │       └─ todo.md          │        │         │
  │          (全是 [ ])       │        │         │
  │                           │        │         │
  └─────── todo.md 从创建到结束，没有任何命令更新它 ──────→

目标 — /verify 作为唯一收口：
/spec → /plan → /build → /test → /review → /verify → /ship
  │       │        │         │        │         │         │
  │       │        │         │        │  +偏差记录  │         │
  │       │        │         │        │         │         │
  │       │        │         │        │         ├─ 读: git log（唯一可信）
  │       │        │         │        │         │      + plan.md 检查点定义
  │       │        │         │        │         │      + review 偏差记录
  │       │        │         │        │         │      + npm test/build 结果
  │       │        │         │        │         │
  │       │        │         │        │         ├─ 反推: 从提交历史重建实际完成状态
  │       │        │         │        │         │
  │       │        │         │        │         ├─ 重写: todo.md（首次也是唯一一次更新）
  │       │        │         │        │         │      plan.md（追加执行偏差章节）
  │       │        │         │        │         │
  │       │        │         │        │         └─ 产出: 验证报告
```

### 各命令的增量变更

| 命令 | 变更 | 工作量 |
|---|---|---|
| `/build` | **步骤 8 修正** — 移除「标记任务为完成」这句从未实现的死文字。`/build` 专心做增量实现和提交，任务状态由 `/verify` 统一反推 | 删 1 行 |
| `/review` | **小改** — 输出末尾追加结构化「偏差记录」块：发现的 bug、需要返工的任务 ID、执行中新增的任务 | +10 行模板 |
| `/verify` | **全新** — 四阶段收口命令。不信任 todo.md 当前状态，从 git log 反推 | ~80 行命令定义 |
| `git-workflow-and-versioning` | **修正触发** — 描述从「任何代码更改时」改为「由 /build 和 /verify 内部调用；在提交边界、分支操作、冲突解决时触发」 | 改 1 行 description |
| `/ship` | **无变更** — 继续并行审查 + 发布决策 | 0 |

## `/verify` 命令设计

### 四个阶段

**阶段 1：采集信号**（不信任 todo.md 当前状态）

```
输入源：
├── docs/plans/<feature>/plan.md    → 原始计划、检查点定义（只读——提取「应该完成什么」）
├── docs/plans/<feature>/todo.md    → 任务列表骨架（只读——提取任务ID和描述，忽略 [ ]/[x] 状态）
├── git log --oneline (计划创建至今) → ★ 唯一可信的执行事实——从提交消息反推任务完成事件
├── git diff (计划创建时的HEAD..当前HEAD) → 实际变更的代码
├── /review 输出中的偏差记录         → bug、返工、新增任务（补充 git log 无法体现的信息）
├── npm test 结果                   → 当前测试状态
└── npm run build 结果              → 当前构建状态
```

**为什么以 git log 为唯一可信源：** `/build` 的步骤 8 从未被执行，todo.md 的 `[x]` 标记不可信。但 `/build` 确实做了提交——每完成一个增量就 commit 一次。git 历史是不会撒谎的执行记录。

**阶段 2：从 git log 反推任务完成状态**

反推逻辑：
1. 读取 plan.md 获取每个任务的描述和涉及文件
2. 读取 git log 获取所有提交消息和变更文件
3. 用提交消息中的关键词 + 变更文件路径，将每个提交映射到对应的 plan 任务
4. 有匹配提交的任务 → ✅ 完成；无匹配提交的任务 → ❌ 待确认
5. 有提交但对不上任何原始任务 → ➕ 执行中新增

对每个检查点和每个任务，判定状态：

| 判定 | 条件 | 标记 |
|---|---|---|
| ✅ 完成 | git log 中存在匹配该任务的提交 + 涉及文件吻合 | `[x]` |
| ⚠️ 部分 | 有相关提交但未覆盖任务描述的全部范围 | `[~]` 附说明 |
| ❌ 未完成 | git log 中无匹配提交 / 检查点条件不满足 | `[ ]` |
| ➕ 新增 | git log 中存在提交但对不上任何原始任务 | `[+]` 附来源 |
| ➖ 废弃 | 原始计划中的任务，git log 和代码变更中均无痕迹 | `[-]` 附原因 |

**阶段 3：产物同步**

重写两个产物文件：

- **todo.md：** 按偏差分析结果重写所有任务状态。新增任务追加到对应阶段。废弃任务保留但标记为 `[-]`。
- **plan.md：** 检查点更新为实际验证结果。追加 `## 执行偏差` 章节，记录所有与原始计划的偏离及原因。

**阶段 4：验证报告**

```markdown
## 验证报告：[功能名称]

### 检查点结果
| 检查点 | 状态 | 备注 |
|---|---|---|
| 基础阶段 | ✅ | 3/3 任务完成 |
| 核心功能 | ⚠️ | 任务4因审查返工重做 |
| 完成 | ✅ | 所有验收标准满足 |

### 任务清单同步
- 原始计划: 7 个任务
- 实际完成: 7 个任务（其中 1 个重做，2 个新增，1 个废弃）
- todo.md 已重写

### 计划偏差
- **新增:** 任务 8 — 修复审查发现的 XSS 漏洞（来源: /review）
- **废弃:** 任务 3 — 内联样式方案改为 Tailwind，无需独立样式任务
- **重做:** 任务 4 — 审查发现边界情况遗漏后重写

### 开放项
- [ ] 性能测试未覆盖（plan.md 检查点3 性能验证 — 需在 /ship 阶段处理）
- [ ] 文档更新未提交（CHANGELOG.md）

### 就绪判定
✅ 可以进入 /ship
```

## 需验证的关键假设

- [ ] **git log 的提交消息包含足够信息来反推任务完成事件** —— 抽查 3 个历史 `/build` 产出的提交历史，验证能否从 `feat:` `fix:` `refactor:` 前缀 + 变更文件路径反向映射到 plan 中的任务 ID
- [ ] **匹配准确率可接受** —— 如果多个任务涉及同一个文件，仅靠提交消息可能无法精确区分。需验证误匹配率是否在可接受范围（目标是 >90% 正确映射）
- [ ] **`/review` 能产出可解析的偏差记录** —— 用 2 个真实审查场景测试偏差记录模板，确认 agent 能稳定产出结构化记录而非自由文本

## MVP 范围

- `/verify` 命令定义文件（`.claude/commands/verify.md`）
- 四个阶段的完整实现（重点是阶段 2 的 git log 反推逻辑）
- `/build` 步骤 8 死文字移除（1 行删除）
- `/review` 偏差记录模板追加（10 行以内）
- `git-workflow-and-versioning` 触发条件修正（1 行）

## 不做（及原因）

- **不让 `/build` 更新 todo.md** — 尝试过（步骤 8），从未实现。与其修复分散的标记逻辑，不如让 `/verify` 集中处理。单一职责。
- **不修改 `/ship`** — `/ship` 专注发布决策，不需要理解计划产物。`/verify` 是它的前置步骤。
- **不做全自动后台同步** — 每个命令自动更新产物的方案（变体 3）架构脆弱。保留一个显式的、单一职责的收口步骤。
- **不做 worktree 强制隔离** — `/verify` 主要是读操作 + 两个文件写入。worktree 留作未来并行场景的可选项。
- **不创建新的产物格式** — 继续使用现有 plan.md + todo.md 格式。plan.md 追加 `## 执行偏差` 章节。
