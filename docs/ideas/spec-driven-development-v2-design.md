# spec-driven-development v2 设计

## 问题陈述

**我们如何能**将 brainstorming 技能中的有效流程模式（硬门控、9 步清单、流程图、自查、用户审查关口、项目拆分）吸收到 spec-driven-development 中，同时保留其独有优势（显式列出假设、成功标准重构、6 核心领域模板），使规范流程更严谨、更防跳过？

## 推荐方向：以 brainstorming 为底座，注入 spec 独有优势

brainstorming 的流程骨架（9 步编号清单 + Graphviz 流程图 + `<硬门控>` 标签）已被证明是有效的防跳过架构。spec-driven-development 的 6 核心领域模板、显式列出假设、成功标准重构是经过验证的规范产出物格式。将两者融合：

- 流程骨架用 brainstorming 的 9 步
- 产出模板用 spec 的 6 核心领域
- 假设列表插入在项目上下文探索之后
- 删除不适用于 agent-skills 的视觉辅助引用
- 保留成功标准重构和"保持规范生命力"

## 融合后的 9 步流程

```
1. 探索项目上下文 → 2. 显式列出假设 → 3. 提供视觉辅助 → 4. 提出澄清问题
→ 5. 提出 2-3 种方法 → 6. 分节展示设计 → 7. 编写设计文档
→ 8. 规范自查 → 9. 用户审查 → 过渡到实现
```

### 每步详细设计

| 步骤 | 来源 | 内容 |
|------|------|------|
| 1. 探索项目上下文 | brainstorming | 检查文件、文档、最近提交。评估范围：请求过大时标记并帮助拆分 |
| 2. 显式列出假设 | spec（原有） | **前置至此。** 在编写任何内容前先列出所有假设，格式："我所做的假设：1. ... 2. ... → 请立即纠正我" |
| 3. 提供视觉辅助 | brainstorming | 如果涉及视觉问题，独立消息询问。不适用 agent-skills 则简化 |
| 4. 提出澄清问题 | brainstorming | 一次一个，理解目标/约束/成功标准。偏好选择题。包含 spec 的"成功标准重构"技巧 |
| 5. 提出 2-3 种方法 | brainstorming | 含权衡与推荐。spec 的 6 核心领域在此作为评估框架 |
| 6. 分节展示设计 | brainstorming 骨架 + spec 模板 | 使用 spec 的 6 核心领域模板（目标/命令/结构/风格/测试/边界）+ 成功标准 + 待解决问题 |
| 7. 编写设计文档 | 两者 | 保存至 `docs/specs/YYYY-MM-DD-<topic>-design.md`，提交 git |
| 8. 规范自查 | brainstorming | 占位符扫描 → 内部一致性 → 范围检查 → 歧义检查。内联修复 |
| 9. 用户审查 | brainstorming | 独立关口："规格说明已撰写并提交至 `<路径>`。请审查" |

## 关键变更清单

### 从 brainstorming 吸收

| 元素 | 融合方式 |
|------|---------|
| `<硬门控>` | 加入：在展示设计并获得批准前，不得调用任何实现技能 |
| 9 步编号流程 | 替换当前隐含步骤，作为主骨架 |
| Graphviz 流程图 | 加入，反映融合后的 9 步流程 |
| 项目上下文探索 | 作为第 1 步 |
| 大项目拆分指导 | 在第 1 步中：多子系统 → 帮助拆分为子项目 |
| 规范自查（4 项） | 作为第 8 步：占位符/一致性/范围/歧义 |
| 用户审查关口 | 作为第 9 步：独立关口 + 对话模板 |
| spec-document-reviewer-prompt | 作为引用：可在自查后派发子代理复审 |
| YAGNI 原则 | 融入"提出方法"和"展示设计"步骤 |
| "不做清单"概念 | 加入规范模板 |
| 反模式"这太简单了" | 扩展借口表，加入 brainstorming 的论据 |

### 从 spec 保留并增强

| 元素 | 处理方式 |
|------|---------|
| 显式列出假设 | 前置为第 2 步（紧接上下文探索之后） |
| 6 核心领域模板 | 融入第 6 步"分节展示设计" |
| 成功标准重构 | 融入第 4 步"提出澄清问题" |
| 保持规范生命力 | 保留为独立章节 |
| 三级边界体系 | 保留在模板中 |
| 输出路径 `docs/specs/` | 保持不变 |

### 删除/不适用的

| 元素 | 原因 |
|------|------|
| 视觉辅助工具（visual-companion） | 依赖 Node.js 服务器 + 浏览器，agent-skills 采用 bash-only 脚本体系，不可用。仅保留"是否涉及视觉问题？"的决策点，但给出纯文本替代方案 |
| brainstorming 的输出路径偏好覆盖 | 保持 agent-skills 的 `docs/specs/` 路径 |
| `docs/superpowers/specs/` | 不采用，superpowers 是独立技能系统 |

## 需验证的关键假设

- [ ] brainstorming 的硬门控机制在 agent-skills 中有效——`<硬门控>` 标签依赖的解析框架是否与 agent-skills 的技能加载机制兼容？验证方法：编写 v2 后实际调用 `/spec` 并观察代理是否遵守门控
- [ ] 融合后的技能不超过 500 行——验证方法：写完 v2 后用 `wc -l` 检查
- [ ] 现有引用 spec-driven-development 的其他技能不需要调整——验证方法：grep 所有 SKILL.md 中对 `spec-driven-development` 的交叉引用

## MVP 范围

仅改造 `skills/spec-driven-development/SKILL.md` 一个文件。不创建新的辅助文件（除非内容超过 100 行且需要拆分）。

包含：
- 9 步流程骨架 + 流程图
- `<硬门控>` 标签
- 假设前置为第 2 步
- 自查清单（第 8 步）
- 用户审查关口（第 9 步）
- 扩展的借口表和危险信号

排除：
- 视觉辅助工具集成（bash 不可用）
- 新建引用文件（除非自查清单超过 100 行）
- 修改其他技能文件
- 修改 `.claude/commands/spec.md`

## 不做（及原因）

- **不创建独立的 spec-reviewer 子代理** —— 当前 agent-skills 已有 code-reviewer 角色可覆盖此需求，brainstorming 的 spec-document-reviewer-prompt 作为参考引用于自查步骤即可
- **不同步修改 brainstorming** —— brainstorming 属于 superpowers 独立技能系统，不属于本次改造范围
- **不修改 `/spec` 命令文件** —— 命令文件只需引用技能名称，技能内容变更对其透明
