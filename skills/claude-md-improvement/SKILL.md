# CLAUDE.md 改进

## 概述

用8条经过实战验证的原则审计现有 CLAUDE.md，逐项打分，产出可执行的改进方案。这8条原则来自真实项目中反复踩坑后的总结——每条都解决一个具体的"Claude 为什么不懂我"的问题。

## 适用场景

- 用户要求检查或改进 CLAUDE.md
- Claude 反复引入不该引入的依赖或模式
- Claude 输出风格与项目规范不一致
- 新项目首次编写 CLAUDE.md
- CLAUDE.md 超过 200 行，需要精简
- 敏感模块（auth / payment / infra）缺少护栏

## 核心流程

### 第 1 步：定位所有 CLAUDE.md 文件

```bash
find . -name "CLAUDE.md" -o -name "*.claude.md" | head -20
```

同时检查是否存在：
- `.claude/settings.json` 中的系统提示词
- `MEMORY.md`（跨会话记忆文件）
- 各子目录下的本地 CLAUDE.md

### 第 2 步：逐条审计（8 项检查）

对每个 CLAUDE.md 文件，逐项打分（通过/部分通过/未通过），记录具体问题位置。

---

**原则 1：长度控制——200 行是硬上限**

```bash
wc -l <CLAUDE.md路径>
```

| 行数 | 判定 |
|------|------|
| ≤ 200 | 通过 |
| 200-300 | 部分通过——需精简 |
| > 300 | 未通过——严重超限 |

检查要点：
- 是否有项目历史叙事、营销文案等非执行信息？
- 是否有可以通过指针（引用其他文件）替代的大段描述？
- 每条信息是否通过了"30 秒测试"——新人在 30 秒内读完能否回答：这是什么产品？技术栈是什么？新代码放哪里？

---

**原则 2：技术栈禁止清单——"不要引入什么"与"要引入什么"同等重要**

检查 CLAUDE.md 中是否包含 `Do NOT introduce` 区块：

- [ ] 是否列出了技术栈（正向）
- [ ] 是否列出了禁用库/模式（负向）——至少 3 条
- [ ] 禁用清单是否有明确的理由（如"已迁移到 X""与 Y 样式冲突"）

没有禁用清单的 CLAUDE.md 是危险的——Claude 会出于善意引入它"知道"的最优方案，而这个方案可能和你的项目完全冲突。

---

**原则 3：规则可操作性——5 秒内能判断一段代码是否符合**

逐条检查 CLAUDE.md 中的每条规则：

| 模糊规则（不合格） | 可操作规则（合格） |
|-------------------|-------------------|
| "写干净的代码" | "使用 named export（路由文件除外）" |
| "保持简洁" | "单个组件不超过 200 行" |
| "注重性能" | "图片使用 Next/Image，禁止原生 `<img>`" |

判定标准：读完这条规则后，你能不能在 5 秒内判断一段代码是否符合它？能——合格。不能——标记为需改写。

---

**原则 4：指针而非图书馆——CLAUDE.md 不存知识，存"去哪找"**

检查是否存在以下反模式：

- [ ] CLAUDE.md 中是否有超过 10 行的架构描述？（应该是指针：`docs/architecture.md`）
- [ ] CLAUDE.md 中是否有完整的 API 文档？（应该是指针：`docs/api.md`）
- [ ] 是否定义了上下文分层（Tier 1/2/3）？

期望的 Tier 结构：
```markdown
## Context Tiers
- Tier 1（每次加载）：CLAUDE.md——项目是什么 + 怎么工作
- Tier 2（按需加载）：docs/architecture.md, docs/api.md——工作时自动读取
- Tier 3（忽略）：docs/archive/——除非明确要求，不碰
```

---

**原则 5：敏感模块护栏——本地 CLAUDE.md**

检查以下目录是否有本地 CLAUDE.md：

- `src/auth/` 或 `src/authentication/`
- `src/payment/` 或 `src/billing/`
- `infra/` 或 `terraform/` 或 `deploy/`
- 任何涉及数据库 migration 的目录

每个本地 CLAUDE.md 应包含：
- 安全红线（绝不修改什么）
- 已知陷阱（依赖特定实现的原因）
- 变更前置条件（必须跑哪些测试）

---

**原则 6：Hook 驱动执行——规则变成 Hook 才是强制执行**

检查 CLAUDE.md 中的规则是否与 Hook 配置联动：

```bash
ls .claude/hooks/ 2>/dev/null || echo "无 Hook 配置"
```

- [ ] 格式化规则 → 是否有对应的 PreToolUse hook（如 prettier）？
- [ ] 测试规则 → 是否有对应的 PostToolUse hook？
- [ ] 敏感目录保护 → 是否有阻止直接编辑的 hook？

写在 CLAUDE.md 里的规则是"请记住"；配了 Hook 的规则是"你必须"。

---

**原则 7：MEMORY.md 回路——一个文件实现跨会话记忆**

检查是否存在 MEMORY.md 及 CLAUDE.md 中是否包含记忆指令：

- [ ] CLAUDE.md 中是否有 MEMORY.md 的读取/写入指令？
- [ ] MEMORY.md 文件是否存在？

期望的指令格式：
```markdown
## Memory
`MEMORY.md` 记录了之前任务中发现的关键洞察、最佳实践和已知陷阱。
每次新任务开始前，先读取 MEMORY.md。
每次任务结束后，如果有新的发现，更新 MEMORY.md。
```

---

**原则 8：工作风格——让 CLAUDE.md 承载你的工作方式**

检查 CLAUDE.md 是否包含工作风格声明：

- [ ] 是否说明了"先给方案还是直接写代码"？
- [ ] 是否说明了沟通语言（如"回复用中文，代码注释用英文"）？
- [ ] 是否说明了讨厌的交互模式（如"不要用 'Great question!' 这类废话"）？
- [ ] 是否说明了自主权限边界（如"重大变更前先问，小优化可以直接执行"）？

---

### 第 3 步：生成审计报告

汇总所有 CLAUDE.md 文件的审计结果，生成结构化报告：

```markdown
## CLAUDE.md 审计报告

### 总览

| 文件 | 行数 | 原则1 | 原则2 | 原则3 | 原则4 | 原则5 | 原则6 | 原则7 | 原则8 | 总评 |
|------|------|-------|-------|-------|-------|-------|-------|-------|-------|------|
| ./CLAUDE.md | 120 | ✅ | ❌ | ⚠️ | ✅ | ❌ | ❌ | ❌ | ⚠️ | 47% |

### 逐条详情

#### 原则 2：禁止清单 — ❌ 未通过
- **问题**：未找到 `Do NOT introduce` 区块
- **建议**：添加至少 3 条禁用库/模式，附理由

#### 原则 3：规则可操作性 — ⚠️ 部分通过
- **模糊规则（需改写）**：
  - 第42行："保持代码简洁" → 建议改为具体指标
  - 第58行："注重性能" → 建议改为可验证的指标

（...逐条列出所有问题及具体行号...）

### 优先修复清单

按影响从高到低排列：
1. [ ] 原则 2：添加技术栈禁止清单（防止引入不兼容依赖）
2. [ ] 原则 3：改写模糊规则为可操作指令
3. [ ] 原则 5：为敏感模块添加本地 CLAUDE.md
4. [ ] 原则 7：建立 MEMORY.md 跨会话记忆回路
```

### 第 4 步：执行改进

按优先修复清单逐项执行改进：

1. **精简**：删除叙事、营销文案、可通过指针替代的大段描述
2. **添加禁止清单**：列出项目已禁用/已迁移的库和模式，每条附理由
3. **改写模糊规则**：每条规则改写为"5 秒内可判断"的具体指令
4. **转换知识为指针**：大段架构/API 描述改为文件引用
5. **创建本地 CLAUDE.md**：为敏感模块创建护栏文件
6. **配置 Hook 联动**：将关键规则与 `.claude/hooks/` 配置关联
7. **建立 MEMORY.md 回路**：在 CLAUDE.md 中添加记忆指令，创建 MEMORY.md
8. **补充工作风格**：添加沟通偏好和自主权限边界

每次编辑后，重新计算行数确保不超过 200 行。

### 第 5 步：验证改进效果

改进完成后，运行验证：

```bash
# 1. 行数检查
wc -l CLAUDE.md

# 2. 检查禁止清单是否存在
grep -i "do NOT introduce\|不要引入\|禁用" CLAUDE.md || echo "❌ 缺少禁止清单"

# 3. 检查 MEMORY.md 引用
grep -i "MEMORY.md" CLAUDE.md || echo "❌ 缺少 MEMORY.md 引用"

# 4. 检查敏感模块护栏
for dir in auth payment billing infra deploy; do
  if [ -d "$dir" ]; then
    [ -f "$dir/CLAUDE.md" ] || echo "⚠️ $dir/ 缺少本地 CLAUDE.md"
  fi
done
```

## 常见合理化借口

| 借口 | 实际情况 |
|------|---------|
| "信息越多越好，Claude 会自己筛选" | 研究表明，过多指令会降低表现。200 行是经过验证的硬上限。你写的每一行多余内容都在挤占 Claude 理解你代码的空间。 |
| "等 Claude 引入错误了我再纠正" | 预防成本远低于修复。一次引入不兼容依赖的错误可能导致后续 10 次会话都在修兼容性问题。 |
| "写干净的代码'这条规则够了" | Claude 不懂"干净"。它懂"用 named export"、"组件不超过 200 行"、"async/await 不用 then 链"。模糊规则对 AI 来说等于没说。 |
| "敏感模块我自己会注意" | 你不会每次都注意。一次对 `src/auth/` 的误操作可能造成安全漏洞。本地 CLAUDE.md 是给你自己上的保险。 |
| "Hook 太复杂，先写规则就行" | Claude 的记忆不可靠。没有 Hook 的规则只是建议。格式化不配 prettier hook，Claude 每次都可能写出不同风格的代码。 |
| "MEMORY.md 没必要，每次会话我都重新交代" | 每次手动开场白浪费的是你的时间和 Claude 的上下文窗口。一个 MEMORY.md 文件省掉每次会话的前 5 条消息。 |

## 危险信号

- CLAUDE.md 超过 300 行且包含大量叙事性文字
- Claude 反复引入已被项目禁用的库或模式
- 规则使用了"干净""简洁""好""性能"等无法 5 秒判定的形容词
- CLAUDE.md 中有完整的架构文档内容（而非指向文档的路径）
- 敏感目录（auth/billing/infra）没有本地 CLAUDE.md
- 有格式化/测试规则但没有对应的 Hook 配置
- 每次新会话都需要手动交代项目背景和工作风格
- 新成员阅读 CLAUDE.md 后无法在 30 秒内回答"这是什么产品？技术栈？新代码放哪？"

## 验证

改进完成后确认：

- [ ] 每个 CLAUDE.md 文件 ≤ 200 行
- [ ] 技术栈区块同时包含"使用"和"禁止引入"两部分
- [ ] 每条代码规则可通过"5 秒测试"——非此即标记为需改写
- [ ] 超过 10 行的描述性内容已转换为文件指针
- [ ] 敏感模块目录存在本地 CLAUDE.md 护栏文件
- [ ] 关键规则（格式化/测试/目录保护）有对应的 Hook 配置
- [ ] CLAUDE.md 包含 MEMORY.md 的读取/写入指令
- [ ] CLAUDE.md 包含工作风格声明（沟通语言/自主边界/讨厌模式）
- [ ] 无关联想、陈旧历史、营销文案等非执行性内容
