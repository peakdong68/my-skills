# 规划与任务拆解

## 概述

将工作分解为小型、可验证的任务，并附上明确的验收标准。优秀的任务拆解，是智能体能够可靠完成工作，还是产出一团乱麻的分水岭。每个任务都应足够小，以便在单次专注的会话中完成实现、测试和验证。

## 适用场景

- 当你已有规范说明，需要将其拆解为可实现的单元时
- 当一个任务感觉太大或太模糊而无法启动时
- 当工作需要在多个智能体或会话之间并行处理时
- 当需要向人类沟通工作范围时
- 当实现顺序尚不明确时

**不适用场景：** 范围明显的单文件修改，或规范说明中已包含良好定义的任务时。

## 规划流程

### 步骤 0：Spec 质量前置检查

### 步骤 1：在编写任何代码之前，先以只读模式操作：

- 阅读规范说明和相关的代码库部分
- 识别现有的模式和约定
- 梳理组件之间的依赖关系
- 记录风险和未知因素
- 如果步骤 0 发现的 spec 问题未解决，在"风险及缓解措施"中记录它们

**在规划期间不要编写代码。** 输出物是一份规划文档，而非实现。

将经过验证的计划文档保存至：`docs/spec/feature_<date>_<id>_<topic>/plan.md`——计划文档本身就是任务清单。多阶段项目拆分为 `plan-1.md`、`plan-2.md` 等。

(注明：用户对计划产物保存位置的偏好设置将覆盖此默认值)

### 步骤 2：识别依赖图

梳理清楚谁依赖谁：

```
数据库模式
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │               │
    │       │               └── UI 组件
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移
```

实现顺序应遵循依赖图自底向上：先构建基础。

### 步骤 3：垂直切分

与其先构建所有数据库层，再构建所有 API 层，最后构建所有 UI 层——不如一次构建一个完整的功能路径：

**不好的做法（水平切分）：**

```
任务 1：构建整个数据库模式
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接所有部分
```

**好的做法（垂直切分）：**

```
任务 1：用户可以创建账户（注册功能的模式 + API + UI）
任务 2：用户可以登录（认证功能的模式 + API + UI）
任务 3：用户可以创建任务（创建任务的模式 + API + UI）
任务 4：用户可以查看任务列表（查询功能 + API + 列表视图 UI）
```

每个垂直切片都能交付可运行、可测试的功能。

### 步骤 4：编写任务

每个任务按 `assets/plan-template.md` 中定义的结构编写——必须先读取模板文件。任务结构包含以下组成部分，缺一不可：

1. **背景（三要素）** — `[业务语境]`（此 Task 在功能中的位置）、`[修改原因]`（为什么需要这个变更）、`[上下游影响]`（依赖哪些前置 Task，输出被哪些后续 Task 依赖）
2. **涉及文件** — 区分"新建"和"修改"，用表格列出每个文件的路径
3. **执行步骤** — 每个步骤包含：位置（文件路径+行号）、操作（LLM 可直接执行的指令，不是泛化描述）、原因（为什么这么做）
4. **检查步骤** — 每个检查包含：具体 shell 命令 + 可验证的预期输出。不允许"手动检查"等模糊措辞
5. **认知变更（如适用）** — 需要记录到 CLAUDE.md 中的持久化信息（新增导出函数、架构变更等）

不含"认知变更"的任务仍属有效，但缺少其他四个组成部分的任何一个都视为不合格。

### 步骤 5：排序与检查点

按以下方式排列任务：

1.  依赖关系得到满足（先构建基础）
2.  每个任务执行完毕后，系统都保持可工作的状态
3.  每完成 2-3 个任务后，设置验证检查点
4.  高风险任务排在前面（尽早失败）

添加明确的检查点：

```markdown
## 检查点：完成任务 1-3 之后

- [ ] 所有测试通过
- [ ] 应用可以无错误地构建
- [ ] 核心用户流程可以端到端跑通
- [ ] 继续前需经人类审查
```

## 任务粒度指南

| 大小     | 文件数 | 范围                      | 示例                   |
| -------- | ------ | ------------------------- | ---------------------- |
| **超小** | 1      | 单一函数或配置更改        | 添加一个验证规则       |
| **小**   | 1-2    | 一个组件或端点            | 添加一个新的 API 端点  |
| **中**   | 3-5    | 一个功能切片              | 用户注册流程           |
| **大**   | 5-8    | 多组件功能                | 带筛选和分页的搜索功能 |
| **超大** | 8+     | **过大 — 需要进一步拆解** | —                      |

如果一个任务是大或超大，应将其拆分为更小的任务。智能体处理小和中任务时表现最佳。

**何时应进一步拆解任务：**

- 预计花费超过单次专注会话的时间（大约 2 小时以上的智能体工作时长）
- 无法在 3 个或更少的要点内描述清楚验收标准
- 涉及两个或以上独立的子系统（例如，认证和计费）
- 发现自己在任务标题中使用了“和”字（这是两个任务的信号）

## 计划产物

将经过验证的计划文档保存至 `docs/spec/feature_<date>_<id>_<topic>/plan.md`（多阶段项目使用 `plan-1.md`、`plan-2.md` ...）。文档结构遵循 `assets/plan-template.md`——在撰写前必须先读取模板文件。计划文档本身就是任务清单（含所有 [ ] checkbox）。

文档保存后立即 `git add` + `git commit -m "plan: add <topic> 执行计划"` 提交。

### 计划自查

**使用 `Agent` 工具分派独立子代理执行审查`plan-checker`**。子代理以对抗立场逐任务检查三层：L1 结构完整性（五字段齐全 / 粒度 / 检查点 / 依赖顺序）→ L2 需求覆盖（spec 目标→任务映射 / 垂直切片 / 不做列表对齐）→ L3 执行鲁棒性（风险 / 并行化 / 回滚安全 / 验证可运行性），产出结构化审查报告。作者不得自查——独立审查才能发现盲点。

审查报告返回后：

- **✓ 可以继续** → 进入用户审查步骤
- **✗ 严重问题** → 修复计划后重新分派子代理审查，通过后再进入用户审查

<HARD-GATE>
计划保存并提交后，回复中只需做三件事——**展示成果、邀请审阅、指明下一步**，此外不得追加任何其他内容（不追问、不提议、不催促）：

> "计划文档已保存至 `docs/spec/feature_<date>_<id>_<topic>/plan.md`（多阶段则为 plan-<N>.md）。请审查任务顺序、依赖关系和验收标准。确认无误后请执行 **/build** 开始增量实现。"

用户反馈分两条路径处理：

- **用户有问题或建议** → 根据反馈修改计划、重新保存并提交，然后重复上述审查提示，直到用户确认无误
- **用户明确表示"没问题"或"可以开始"** → 输出：

> "计划已确认。请执行 **/build** 命令开始增量实现（逐任务测试 → 实现 → 验证 → 提交）。"

**绝对禁止：用户未明确确认前，不得开始任何代码实现。** 跳过用户确认直接写代码，等于把未经甲方签字的施工图直接浇筑成混凝土。
</HARD-GATE>

## 并行化机会

当有多个智能体代理或会话可用时：

- **可安全并行：** 独立的功能切片、针对已实现功能的测试、文档
- **必须顺序执行：** 数据库迁移、共享状态的变更、依赖链
- **需要协调：** 共享同一 API 契约的功能（先定义契约，再并行实现）

## 常见借口辨析

| 借口                         | 现实                                                                 |
| ---------------------------- | -------------------------------------------------------------------- |
| “我边做边想”                 | 这就是你会搞出一团乱麻、需要返工的原因。花 10 分钟规划能节省数小时。 |
| “任务显而易见”               | 还是要写下来。明确的任务能暴露隐藏的依赖和被遗忘的边界情况。         |
| “规划是额外开销”             | 规划本身就是任务。没有计划的实现只是在打字。                         |
| “我能把所有东西都记在脑子里” | 上下文窗口是有限的。书面的计划可以在会话边界和压缩后依然存在。       |

## 危险信号

- 没有书面任务列表就开始实现
- 任务只写了“实现某某功能”而没有验收标准
- 计划中没有验证步骤
- 所有任务都是超大规模的
- 任务之间没有检查点
- 未考虑依赖顺序

## 验证

在开始实现前，确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖关系已确认且排序正确
- [ ] 没有任何任务涉及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已经审查并批准了该计划
