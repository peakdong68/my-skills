# 工作流收口验证

## 概述

闭合计划-执行反馈回路。`/build` 的步骤 8 会逐任务标记 `[x]`，但执行中的演变（审查返工、新增任务、计划偏离）仍无法写回。结果：plan.md 中的 checkbox 状态**不完全可信**——有真标记也有遗漏。

该技能是最终收口验证——**从 git log（唯一可信的执行记录）完全反推实际完成情况**，核验并更正 plan.md 中的任务状态，产出验证报告。

三种演变各有检测机制：**审查返工** → 解析 `/review` 偏差记录表格（REWORK 条目）；**新增任务** → git log 中有提交但对不上任何原始计划任务（标记 `[+]`）；**计划偏离** → 原始计划任务在 git log 中无对应提交（标记 `[-]`），或执行顺序/文件范围与计划不符。三者最终汇入 `## 执行偏差` 章节统一呈现。

## 适用场景

- `/review` 完成、`/ship` 之前，作为收口步骤执行
- `docs/spec/feature_<date>_<id>_<topic>/plan.md` 存在时
- 计划执行中发生了演变（审查修复、新增任务、计划变更），需要同步产物

**不适用场景：**

- 无计划产物（需先运行 `/planning`）
- 计划创建后无任何提交（无所验证）

## 流程

### 阶段 1：采集信号

不完全信任计划中 checkbox 的当前 `[ ]/[x]` 状态——`/build` 可能已标记部分任务，但遗漏和未标记的偏离只能从 git 中确认。从以下信息源重建实际状态：

**1.1 定位计划产物**

```
- 读取 `docs/spec/feature_<date>_<id>_<topic>/plan.md`（多阶段使用 `plan-<N>.md`） — 计划与检查点定义 + 任务 checkbox 骨架（忽略现有 `[ ]/[x]` 状态，仅提取任务 ID 和描述）
如果找不到，列出可用的 plan.md/spec-design.md 供用户选择。
```

**1.2 检查已有验证报告**

这是幂等性关键步骤——`/sdd-verify` 可能被多次执行（审查返工后重新验证）。

```
检查 docs/spec/feature_<date>_<id>_<topic>/verify-report.md 是否存在。
```

两种情况：

| 场景                       | 行为                                                                                                                                        |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **首次验证**（文件不存在） | 正常执行全部四个阶段，生成首份 `verify-report.md`，`验证序号` 记为 1                                                                        |
| **重复验证**（文件已存在） | 读取旧报告，提取：上次就绪判定、任务状态快照、检查点结果、开放项列表。本次验证结果与上次做 diff 对比，在报告中追加 `## 与前次验证对比` 章节 |

对于重复验证，关键判断：

- **如果 git HEAD 与上次验证时完全相同**（无新提交、无新未暂存变更）→ 报告标注"与上次验证无差异，可跳过"，但仍产出新报告（覆盖旧报告，更新生成时间）
- **如果有新提交** → 仅对新提交覆盖的任务重新判定，其余任务沿用上次判定结果（除非新提交涉及跨任务影响）
- **如果上次报告的开放项全部关闭** → 在对比章节中标注"上次开放项已清零"

**1.3 读取原始计划定义**

- 从 plan.md 提取：任务列表（ID、描述、验收标准、涉及文件）、检查点定义（每阶段的条件）
- 从 plan.md 提取任务列表和 checkbox 状态——忽略所有 `[ ]/[x]` 状态标记（不可信），仅使用任务 ID 和描述骨架

**1.4 确定 git 时间范围**

★ 此步骤涉及 git 历史查询，先调用 `git-workflow-and-versioning` 技能以获取提交消息格式规范（`feat:`/`fix:`/`refactor:` 等类型前缀）和 `git log` 最佳实践。

```bash
# 找到计划文件的首次创建提交
git log --diff-filter=A --follow --format="%H" -- docs/spec/feature_<date>_<id>_<topic>/plan.md

# 从计划创建至今的提交范围（使用 git-workflow-and-versioning 定义的提交格式解析）
git log --oneline --name-status <first-commit>..HEAD
```

如果 plan.md 未被 git 追踪（新文件），则使用 `git log --oneline --name-status HEAD` 获取所有提交。

**1.5 提取 git 执行记录**

从每条提交中提取：

- 提交类型（`feat:` / `fix:` / `refactor:` / `test:` / `docs:` / `chore:`）
- 变更文件路径
- 提交消息描述

**1.6 读取 `/review` 偏差记录**

从 `/review` 输出中解析 `### 偏差记录 (Deviation Record)` 章节的 markdown 表格。提取 NEW/REWORK/SKIP 条目。

如果偏差记录表格缺失，标注为数据质量问题（非致命错误）并继续。

**1.7 收集当前代码状态**

★ 调用 `git-workflow-and-versioning` 以获取工作区状态检查规范（`git status`、`git diff` 解读模式）。

- `git status` — 是否有未提交的变更
- `git diff` — 未暂存变更的具体内容（配合 git-workflow-and-versioning 的 diff 审查模式）
- 检查项目是否有 `npm test` 脚本：若有则运行并记录结果
- 检查项目是否有 `npm run build` 脚本：若有则运行并记录结果
- 如果项目没有这些脚本，报告"项目无可执行测试/构建"并跳过

### 阶段 2：从 git log 反推任务完成状态

这是核心算法。不完全信任计划中的 checkbox 状态——从提交历史核验并纠正真实情况。

**2.1 提交→任务映射**

对于 plan.md 中定义的每个任务：

1. 提取任务描述关键词（标题中的动作词 + 涉及的实体名）
2. 提取 `可能涉及的文件` 列表
3. 扫描每条 git log 条目，计算匹配强度：
   - **强匹配**：提交消息关键词命中 + 变更文件路径与涉及文件有交集
   - **弱匹配**：仅文件路径有交集（消息关键词无命中）
   - **无匹配**：关键词和文件路径均不匹配

**2.2 映射示例**

```
提交: "feat: 添加任务创建表单组件"
变更文件: src/components/TaskCreateForm.tsx, tests/components/TaskCreateForm.test.tsx
计划任务: "任务 3: 创建任务表单 UI（涉及文件: src/components/TaskCreateForm.tsx）"

→ 强匹配: 关键词 "任务创建表单" + 文件交集 TaskCreateForm.tsx
→ 判定: [x] 完成
```

**2.3 任务状态判定**

对每个任务，根据映射结果判定状态：

| 判定   | 标记  | 条件                                                     |
| ------ | ----- | -------------------------------------------------------- |
| 完成   | `[x]` | 存在强匹配提交，覆盖任务描述的全部范围                   |
| 部分   | `[~]` | 存在弱匹配，但未覆盖全部范围（附说明）                   |
| 未完成 | `[ ]` | 无任何匹配提交                                           |
| 新增   | `[+]` | git log 中有提交但对不上任何原始计划任务（附来源）       |
| 废弃   | `[-]` | 原始计划中的任务，git log 和代码变更中均无痕迹（附原因） |

**2.4 检查点验证**

对 plan.md 中定义的每个检查点：

- 读取检查点条件（如"测试通过，构建干净"）
- 检查该阶段所有任务的状态
- 如果所有任务完成且条件满足 → ✅ 通过
- 如果部分条件满足 → ⚠️ 部分通过（记录具体缺口）
- 如果关键条件不满足 → ❌ 未通过

### 阶段 3：产物同步

**3.1 更新 plan.md 中的任务状态**

完全重写（不是选择性更新），确保文件一致性：

- 保留原始任务结构（阶段分组、检查点位置）
- 将每个任务的 `[ ]` 替换为实际判定标记 `[x]` `[~]` `[+]` `[-]`
- 新增任务追加到对应阶段末尾，标注来源
- 废弃任务保留原位置但标记为 `[-]`，添加原因说明

```
示例:
- [x] 任务 1: 数据库模式设置
- [x] 任务 2: API 端点创建
- [-] 任务 3: 内联样式实现  ← 废弃（改用 Tailwind）
- [+] 任务 8: XSS 漏洞修复  ← 新增（来源: /review）
```

**3.2 更新 plan.md**

- 检查点状态更新为实际验证结果（通过/未通过 + 备注）
- 追加 `## 执行偏差` 章节：

```markdown
## 执行偏差

以下记录计划与实际执行之间的所有偏离：

| 任务ID | 偏离类型 | 原始计划     | 实际执行         | 原因             |
| ------ | -------- | ------------ | ---------------- | ---------------- |
| 3      | 废弃     | 内联样式     | 改用 Tailwind    | 审查建议统一方案 |
| 4      | 返工     | 任务列表组件 | 重写边界情况处理 | 审查发现遗漏     |
| 8      | 新增     | --           | XSS 漏洞修复     | 审查发现         |
```

- 更新 `## 风险及缓解措施`（如有新风险）

### 阶段 4：产出验证报告

**4.1 报告文件路径**

将完整验证报告写入 `docs/spec/feature_<date>_<id>_<topic>/verify-report.md`（与 plan.md 同级目录）。终端输出仅呈现执行摘要 + 就绪判定；完整细节（逐任务判定依据、逐检查点条件验证）全部存入文件，供后续审计和 `/ship` 引用。

**4.2 报告模板**

报告文件包含以下章节，按结构化验收模式编排——每个检查项标注来源、目的、操作步骤和预期输出：

```markdown

**生成时间:** <ISO 日期>
**关联计划:** ./plan.md
**关联设计:** ./spec-design.md（如存在）
**Git 范围:** <first-commit>..HEAD（共 N 条提交）
**验证序号:** <第 N 次验证 — 从 verify-report.md git log 计数>

> 所有验收项均可自动化验证。本报告由 `/sdd-verify` 自动生成。
```

**4.3 逐任务验证项生成规则**

对于计划中的每个任务，在报告中生成一个验证条目：

1. **描述** — 从 plan.md 任务标题提取
2. **来源** — plan.md Task ID + 设计文档章节引用（如有）
3. **目的** — 从任务背景描述中提炼为一句
4. **判定依据** — 阶段 2 的提交→任务映射结果（提交 hash + 消息）
5. **操作步骤** — 至少一条自动验证命令（grep 文件存在性 / 测试运行 / 类型检查），标注 `[A]`，包含期望输出模式
6. **结果** — `✅` 通过 / `❌` 未通过 / `⚠️` 部分

若计划任务包含检查步骤（`检查步骤:` 章节），将其中每条命令转换为操作步骤中的 `[A]` 条目。

**4.4 终端输出**

终端仅输出执行摘要（一行）和就绪判定。完整报告路径打印给用户：

```
✅ 验证完成 | 计划 5 / 完成 4 / 新增 1 / 废弃 0 / 返工 0 | 检查点 2/3 通过
⚠️ 部分通过 — 完整报告: docs/spec/feature_<date>_<id>_<topic>/verify-report.md
就绪判定: ❌ 阻塞 — 关键检查点"核心功能"未通过（任务 4 无匹配提交）
```

**4.5 下一步提示**

当验证结论为 **✅ 全部通过** 且就绪判定为 **✅ 可以进入 /ship** 时，在终端输出后追加：

```
✅ 全部验收项通过。请执行 /ship 进入发布前检查。
```

行为规则：

- **全部通过（✅）** → 提示用户执行 `/ship`
- **非全部通过（⚠️ 或 ❌）** → 不提示 `/ship`，报告阻塞原因并由用户决定后续步骤

## 常见合理化借口

| 借口                               | 现实                                                                                                                |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| "plan.md 不需要更新，代码才是真相" | 两份真相比一份好。更新后的 plan.md 是给人类和未来 agent 的结构化索引——没有它，每次都需重读 git log。                |
| "git log 已经够清楚了"             | git log 是线性的、按时间排序的事件流。更新后的 plan.md 是按任务组织的状态快照。两者互补。                           |
| "等全部完成后再更新"               | 经验证明这个"一次性更新"永远不会发生。`/sdd-verify` 就是被设计为这个"一次性"步骤。                                  |
| "手动检查就够了"                   | 跨会话时人类记忆和 agent 上下文都会丢失。书面化的验证报告是持久记录。                                               |
| "提交消息格式可能对不上"           | 比计划中全是 `[ ]` 好不止一个数量级。匹配规则在 2-3 个执行周期后可校准。                                            |
| "终端输出就够了，不需要文件"       | 终端输出随会话消失。`verify-report.md` 是 `/ship` 的输入、审计的痕迹、下次 `/sdd-verify` 的基线。                   |
| "全部通过了，不用归档"             | 已完成的功能目录堆积在 `docs/spec/` 下污染工作区。归档到 `docs/spec/archive/` 保留历史，清空活跃视图。              |
| "归档了目录就行，索引不用更新"     | 目录归档后若不同步更新 `docs/spec/global/index.md`，后续 agent 无法从索引发现已归档 feature，等于归档了但找不回来。 |

## 危险信号

- plan.md 中大量任务 checkbox 仍是 `[ ]` 或与 git log 不符（确认了遗漏——这正是该技能要解决的）
- git log 只有一条 "update" 提交（违反原子提交原则，无法反推粒度）
- plan.md 无法定位或文件缺失（计划产物不完整）
- 代码中存在大量未提交变更（覆盖了部分计划任务但无记录）
- `/review` 输出中无偏差记录表格（审查者跳过了结构化输出）
- 多个任务映射到同一个提交（暗示提交粒度过粗）
- 提交消息没有类型前缀（`feat:` / `fix:` 等），无法提取任务关联

- [ ] plan.md 已定位并读取
- [ ] 已有 `verify-report.md` 检查已完成（首次/重复验证判定，重复时旧报告已读取）
- [ ] git log 从计划创建至今的所有提交已分析
- [ ] 每个原始任务的完成状态已判定，附判定依据
- [ ] 每个检查点的验证结果已记录
- [ ] plan.md 已更新，所有 `[ ]` 更新为 `[x]` 实际状态
- [ ] plan.md 已追加 `## 执行偏差` 章节（如有偏离）
- [ ] 验证报告已写入 `verify-report.md`（与 plan.md 同级目录），包含逐任务判定依据和就绪判定
- [ ] 重复验证时报告包含 `## 与前次验证对比` 章节（任务状态变化、检查点变化、开放项收敛）
