# 增量式实现

## 概述

以薄的垂直切片方式进行构建 —— 实现一个部分，测试它，验证它，然后扩展。避免一次性实现整个功能。每个增量都应使系统处于可工作、可测试的状态。这正是让大型功能变得可管理的执行纪律。

## 适用场景

- 实现任何涉及多个文件的变更
- 基于任务拆解构建新功能
- 重构现有代码
- 任何时候你忍不住想一次写超过 ~100 行再测试

**不适用场景：** 范围已经极小的单文件、单函数变更。

## 增量循环

```
┌──────────────────────────────────────┐
│                                      │
│   实现 ──→ 测试 ──→ 验证 ──┐         │
│       ▲                    │         │
│       └───── 提交 ◄────────┘         │
│              │                       │
│              ▼                       │
│          下一切片                    │
│                                      │
└──────────────────────────────────────┘
```

对每个切片：

1. **实现** — 最小的完整功能片段
2. **测试** — 调用 `test-driven-development`，先写失败测试再实现
3. **验证** — 执行增量检查清单：测试通过 + 构建成功 + 类型检查通过 + 功能验证
4. **提交** — 调用 `git-workflow-and-versioning`，原子提交，描述性消息
5. **标记** — 将 plan.md 中对应任务 checkbox 从 `[ ]` 更新为 `[x]`，记录实际完成内容
6. **进入下一片** — 持续前进，不要重新开始

## 切片策略

### 垂直切片（推荐）

构建一条贯穿整个堆栈的完整路径：

```
切片1：创建任务（数据库 + API + 基本 UI）
    → 测试通过，用户可以通过 UI 创建任务

切片2：列出任务（查询 + API + UI）
    → 测试通过，用户可以看到自己的任务

切片3：编辑任务（更新 + API + UI）
    → 测试通过，用户可以修改任务

切片4：删除任务（删除 + API + UI + 确认）
    → 测试通过，完整 CRUD 完成
```

每个切片都交付可工作的端到端功能。

### 契约优先切片

当后端和前端需要并行开发时：

```
切片0：定义 API 契约（类型、接口、OpenAPI 规范）
切片1a：根据契约实现后端 + API 测试
切片1b：根据匹配契约的模拟数据实现前端
切片2：集成并进行端到端测试
```

### 风险优先切片

首先解决风险最高或最不确定的部分：

```
切片1：证明 WebSocket 连接可行（最高风险）
切片2：在已验证的连接上构建实时任务更新
切片3：添加离线支持和重连
```

如果切片1失败，你会在投入切片2和3之前就发现它。

## 联动技能

增量循环的每个阶段必须联动以下技能，不可跳过：

| 阶段           | 联动技能                      | 说明                                                                  |
| -------------- | ----------------------------- | --------------------------------------------------------------------- |
| **实现**       | —                             | 按计划编写代码，遵循规则 0-5                                          |
| **测试**       | `test-driven-development`     | **必须调用**——先写失败测试，再实现使之通过                            |
| **验证**       | —                             | 执行下方"增量检查清单"：测试通过 + 构建成功 + 类型检查通过 + 功能验证 |
| **提交**       | `git-workflow-and-versioning` | 原子提交——一个切片一个 commit，描述性消息                             |
| **进入下一片** | —                             | 读取 plan.md，定位下一个 `[ ]` 任务，确认前置依赖已满足               |

这些技能不是"可选参考"——增量循环的质量直接取决于每个阶段是否执行了对应技能的完整流程。跳过测试直接实现、跳过验证直接提交，都会破坏增量式构建的核心前提。

## 实现规则

### 规则0：简单至上

在编写任何代码之前，先问自己：“最可行的最简单方案是什么？”

编写代码后，对照以下检查重新审视：

- 能否用更少的行数完成？
- 这些抽象是否对得起它们的复杂性？
- 一位资深工程师看到这些代码，会不会说“你为什么不直接……”？
- 我是在为假设的未来需求构建，还是为了当前的任务？

```
简洁性检查：
✗ 为单一通知构建带有中间件管道的通用 EventBus
✓ 简单的函数调用

✗ 为两个相似组件使用抽象工厂模式
✓ 两个具有共享工具的直截了当的组件

✗ 为三个表单使用配置驱动的表单构建器
✓ 三个表单组件
```

三行相似的代码比过早的抽象要好。先实现天真、明显正确的版本。只有在通过测试证明其正确性之后，才进行优化。

### 规则0.5：范围约束

仅触及任务所需的代码。

禁止：

- “清理”与你变更相邻的代码
- 重构你未修改文件中的导入
- 删除你不完全理解的注释
- 添加规范中没有的功能，因为它们“似乎有用”
- 在你只是阅读的文件中现代化语法

如果你注意到任务范围之外值得改进的地方，记下来 —— 不要修复它：

```
注意到但未触碰：
- src/utils/format.ts 有一个未使用的导入（与本任务无关）
- 认证中间件可以使用更好的错误消息（单独任务）
→ 是否要我为此创建任务？
```

### 规则1：一次只做一件事

每个增量只改变一件逻辑上的事。不要混合关注点：

**不佳实践：** 一次提交同时添加一个新组件、重构一个现有组件并更新构建配置。

**良好实践：** 三次独立的提交 —— 每次对应一项改动。

### 规则2：保持可编译状态

每次增量之后，项目必须能够构建，并且现有测试必须通过。不要在切片之间让代码库处于不可用状态。

### 规则3：为未完成的功能使用功能标志

如果某个功能尚未准备好面向用户，但你需要合并增量时：

```typescript
// 用于进行中工作的功能标志
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // 新的分享 UI
}
```

这让你可以将小的增量合并到主分支，而不会暴露未完成的工作。

### 规则4：安全默认值

新代码应默认采用安全、保守的行为：

```typescript
// 安全：默认禁用，主动选择启用
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### 规则5：易于回滚

每个增量都应能独立回滚：

- 增加式的变更（新文件、新函数）易回滚
- 对现有代码的修改应最小化且集中
- 数据库迁移应有相应的回滚迁移
- 避免在一个提交中删除某物，并在同一个提交中替换它 —— 应将它们分开

## 与智能体协作

当引导一个智能体进行增量实现时：

```
让我们实现计划中的任务 3。

仅从数据库模式变更和 API 端点开始。
暂时不要触及 UI —— 我们将在下一个增量中处理。

实现后，运行 `npm test` 和 `npm run build` 来验证
没有东西被破坏。
```

要明确每个增量范围内包含什么，不包含什么。

## 增量检查清单

每个增量完成后，请验证：

- [ ] 该改动只做了一件事，并且做完整了
- [ ] 所有现有测试依然通过（`npm test`）
- [ ] 构建成功（`npm run build`）
- [ ] 类型检查通过（`npx tsc --noEmit`）
- [ ] 代码格式检查通过（`npm run lint`）
- [ ] 新功能按预期工作
- [ ] 在 TodoWrite 中将任务标记为完成
- [ ] 该改动已用描述性消息提交（git commit）

## 常见借口

| 借口                           | 现实                                                                        |
| ------------------------------ | --------------------------------------------------------------------------- |
| “我最后再一起测试”             | 错误会叠加。切片1中的一个错误会使切片2-5全部错误。应测试每个切片。          |
| “一次性做完更快”               | 这会让你*感觉*更快，直到某处出错，而你无法从修改的500行中定位是哪行引起的。 |
| “这些改动太小，不值当单独提交” | 小的提交是免费的。大的提交隐藏错误并使回滚痛苦。                            |
| “我稍后再添加功能标志”         | 如果功能未完成，就不应该对用户可见。现在就加上标志。                        |
| “这次重构很小，可以包含在内”   | 重构与功能混合在一起会让审查和调试都变得困难。应将它们分开。                |

## 危险信号

- 编写超过100行代码而未运行测试
- 单次增量中包含多个不相关的改动
- “让我再快速把这个也加上”式的范围扩张
- 为了更快而跳过测试/验证步骤
- 增量间构建或测试失败
- 大量未提交的改动累积
- 在第三个用例要求之前构建抽象
- 在做任务时顺手触碰范围外的文件
- 为一次性操作创建新的实用工具文件

## 验证

完成任务的所有增量后：

- [ ] 每个增量都已单独测试并提交（git commit）
- [ ] 完整测试套件通过
- [ ] 构建是干净的
- [ ] 功能按照规范端到端工作
- [ ] 任务标记为完成 — plan.md 中对应任务 checkbox 是否从 `[ ]` 更新为 `[x]`
- [ ] 没有未提交（git commit）的改动遗留
