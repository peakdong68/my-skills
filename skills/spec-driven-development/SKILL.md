# 规范驱动开发

## 概述

在编写任何代码之前，先撰写结构化的规范。该规范是你与人类工程师共享的唯一事实来源——它定义了我们正在构建什么、为什么构建，以及如何判断是否完成。无规范编码就等同于盲目猜测。

将经过验证的设计规范文档写入 `docs/specs/YYYY-MM-DD-<topic>-design.md`。用户确认后，下一步实施前使用git commit 提交。

## 适用场景

- 启动新项目或新功能
- 需求含糊不清或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 任务实现预计耗时超过 30 分钟

**不适用场景：**单行修复、拼写纠错，或需求明确且自包含的变更。

### 规范制定(SPECIFY)

从高层次愿景开始。向人类提问以澄清细节，直到需求变得具体。

**立即列出所有假设。**在编写任何规范内容之前，先列出你正在采用的假设：

```
我所做的假设：
1. 这是一个 Web 应用（非原生移动应用）
2. 认证使用基于会话的 Cookie（非 JWT）
3. 数据库为 PostgreSQL（基于现有的 Prisma 架构）
4. 目标浏览器仅限现代浏览器（不支持 IE11）
→ 请立即纠正我，否则我将按此进行。
```

不要默默填补含糊不清的需求。规范的全部目的就是在代码编写*之前*暴露误解——而假设是最危险的误解形式。

**撰写一份涵盖以下六个核心领域的规范文档：**

1. **目标(Objective)** —— 我们正在构建什么，为什么？用户是谁？成功是什么样子？

2. **命令(Commands)** —— 带完整参数的可执行命令，而非仅列出工具名称。

   ```
   构建: npm run build
   测试: npm test -- --coverage
   代码检查: npm run lint --fix
   开发: npm run dev
   ```

3. **项目结构(Project Structure)** —— 源代码存放位置、测试存放位置、文档存放位置。

   ```
   src/           → 应用源代码
   src/components → React 组件
   src/lib        → 共享工具库
   tests/         → 单元测试与集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格(Code Style)** —— 用一段真实的代码示例展示你的风格，这比三段文字描述更有效。包括命名规范、格式规则以及良好输出的范例。

5. **测试策略(Testing Strategy)** —— 使用哪个框架、测试文件放在哪里、覆盖率预期、针对不同关注点采用何种测试层级。

6. **边界 (Boundaries)** —— 三级体系：
   - **始终执行：** 提交前运行测试、遵循命名规范、验证输入
   - **先询问：** 数据库架构变更、添加依赖项、更改 CI 配置
   - **禁止：** 提交机密信息、编辑 vendor 目录、未经批准删除失败测试

**规范模板：**

```markdown
# 规范: [项目/功能名称]

## 目标

[我们要构建什么及其原因。用户故事或验收标准。]

## 技术栈

[框架、语言、关键依赖项及其版本]

## 命令

[构建、测试、代码检查、开发 —— 完整命令]

## 项目结构

[附有描述的目录布局]

## 代码风格

[示例代码段 + 关键约定]

## 测试策略

[框架、测试文件位置、覆盖率要求、测试层级]

## 边界

- 始终: [...]
- 先询问: [...]
- 禁止: [...]

## 成功标准

[我们如何判断任务已完成 —— 具体、可测试的条件]

## 待解决问题

[任何需要人类进一步确认的未决事项]
```

**将指令重构为成功标准。**当接收到模糊需求时，将其转化为具体条件：

```
需求："让仪表盘变得更快"

重构后的成功标准：
- 仪表盘在 4G 连接下，LCP < 2.5s
- 首屏数据加载在 500ms 内完成
- 加载期间无布局偏移 (CLS < 0.1)
→ 这些是合适的目标吗？
```

这样你就能向清晰的目标进行循环、重试和求解，而非猜测“更快”到底意味着什么。

## 保持规范的生命力

规范是一份活文档，而非一次性产物：

- **当决策变更时，更新规范文档** —— 如果你发现数据模型需要更改，先更新规范，再实施。
- **当范围变更时，更新规范文档** —— 新增或削减的功能应反映到规范中。
- **提交规范文件** —— 规范应与代码一起纳入版本控制。
- **在 PR 中引用规范** —— 关联到每个 PR 所实现的规范章节。

## 常见借口

| 借口                       | 现实                                                                       |
| -------------------------- | -------------------------------------------------------------------------- |
| "这太简单了，我不需要规范" | 简单的任务不需要*冗长*的规范，但仍然需要验收标准。两行字的规范也是可以的。 |
| "我先写完代码再补规范"     | 那是文档，不是规范。规范的价值在于*编码前*强制澄清问题。                   |
| "写规范会拖慢进度"         | 15 分钟写规范能避免数小时的返工。15 分钟的瀑布式梳理远胜于 15 小时的调试。 |
| "反正需求总是会变的"       | 正因为如此，规范才必须是活文档。一份过时的规范也强过没有规范。             |
| "用户清楚自己想要什么"     | 即使是清晰的请求也带有隐含假设。规范的目的就是暴露这些假设。               |

## 危险信号

- 在没有任何书面需求的情况下开始编写代码
- 在厘清“完成”的定义之前就询问“我是不是应该直接开始构建？”
- 实施任何规范或任务清单中未提及的功能
- 做出架构决策而不记录在案
- 因为“要构建什么显而易见”而跳过规范制定

## 验证

在进入实施阶段之前，请确认：

- [ ] 规范已覆盖所有六个核心领域
- [ ] 人类已审查并批准了该规范
- [ ] 成功标准是具体且可测试的
- [ ] 边界（始终/先询问/禁止）已定义
- [ ] 设计规范文档已保存到代码仓库中的文件
