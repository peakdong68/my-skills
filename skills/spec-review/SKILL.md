# Spec Review

## 概述

在规范文档进入实现计划之前，对其进行独立的质量审查。与规范自查（同实例）不同，此审查**必须由独立子代理执行**——不能是写 spec 的同一个代理。"自己检查自己"等同于没有检查。

审查产出结构化报告，按严重程度分类：严重（强烈建议修复）、重要（应修复）、建议（可选）。最终由用户决定是否进入 plan——此为软门控。

<硬门控>
审查**必须**由独立子代理执行。当本技能被触发时，**必须**分派独立子代理来执行审查规范流程。不得在当前会话中直接执行审查。
</硬门控>

## 何时使用

- 产出设计规范文档后、编写实施计划开始前使用（必经步骤）
- 任何时候需要独立验证规范文档质量
- 用户怀疑 spec 不完整或有遗漏时
- 接手其他人编写的 spec 时

**不适用场景：** 单行修复或配置更改的微规范（不超过一段话的 spec）。

## 审查流程

### 步骤 1：定位 Spec 文档

- 优先使用用户指定的文件路径
- 否则从 `docs/spec/` 下找到最新的 spec 文档（按特征目录名 `feature_*` 排序）
- 如果找不到任何 spec 文档，报告错误并退出
- 读取 spec 文档全文

### 步骤 2：建立需求基线

从以下来源按优先级建立原始需求清单：

1. **对话上下文（高置信度）** —— 派发审查子代理时，将主会话中用户表达的原始需求、澄清问题和确认结论作为上下文传递
2. **idea-refine 文档（中置信度）** —— 如果需求先经过了 idea-refine 流程，检查 `docs/spec/<feature>/idea.md` 下关联的想法文档
3. **Spec 目标章节反推（低置信度）** —— 从 spec 的"目标"章节反推需求清单，标记为低置信度基线

### 步骤 3：范围评估与分批

执行审查前先评估 spec 范围：

- 检查 spec 是否描述了多个独立子系统（如"这个项目包含：聊天系统、文件存储、计费平台"）
- 如果识别到 2+ 个独立子系统，按子系统拆分为单独审查单元
- 每个子系统分别执行 L1/L2/L3 全套检查
- 汇总结论覆盖所有批次

### 步骤 4：三层校验

#### L1 结构完整性

检查 spec 文档的物理结构是否完整：

| #   | 检查项          | 判定标准                                                                                  |
| --- | --------------- | ----------------------------------------------------------------------------------------- |
| 1   | 7 个固定章节    | 需求背景、目标、方案设计、实现要点、验收标准、不做、待解决问题 是否全部存在且有实质性内容 |
| 2   | 方案设计结构    | 是否包含整体架构描述，并拆分了子系统章节（如适用）                                        |
| 3   | 实现要点完整    | 关键技术决策、难点、依赖、新增文件清单、修改文件清单 是否均有内容                         |
| 4   | 文档位置        | 是否在 `docs/spec/` 路径下                                                                |
| 5   | 空章节/占位符   | 是否有章节标题存在但内容为空或仅含 `[占位说明]`、TBD、TODO                                |
| 6   | 验收标准可测试  | 验收标准是否每条能用"是/否"判断，不允许模糊措辞如"体验良好"                               |
| 7   | Feature ID 格式 | 标题是否包含 `YYYY-MM-DD_F00X` 格式的 Feature ID                                          |

#### L2 需求一致性

检查 spec 内容是否覆盖了需求基线中的所有功能点：

| #   | 检查项         | 判定标准                                                                                 |
| --- | -------------- | ---------------------------------------------------------------------------------------- |
| 1   | 功能点覆盖     | 需求基线中的每个功能点是否在 spec 中有对应描述或验收标准                                 |
| 2   | 内部一致性     | 方案设计与实现要点之间是否存在矛盾（如方案设计描述用框架 A 但文件清单列出框架 B 的文件） |
| 3   | 不做清单合理性 | 标记为不做的排除项是否有充分理由，是否与需求基线冲突——避免把核心需求错误标记为不做       |
| 4   | 依赖声明       | 外部依赖（API、数据库、第三方服务）是否已声明                                            |
| 5   | 遗漏检测       | 需求中明确提到的功能是否在 spec 中缺失                                                   |

#### L3 边界与鲁棒性

检查 spec 是否考虑了异常和边界场景：

| #   | 检查项      | 判定标准                                                     |
| --- | ----------- | ------------------------------------------------------------ |
| 1   | 错误处理    | 是否定义了关键操作的错误处理策略（API 失败、超时、网络异常） |
| 2   | 边界输入    | 是否考虑了空输入、超长输入、特殊字符等边界情况               |
| 3   | 空/加载状态 | 是否有空数据状态、加载状态、错误状态的 UI 描述               |
| 4   | 并发与竞态  | 是否提及并发操作或竞态条件的处理                             |
| 5   | 安全考量    | 是否涉及用户输入验证、认证/授权、敏感数据处理                |
| 6   | 回退与降级  | 是否有功能降级或回退策略                                     |

### 步骤 5：产出审查报告

```markdown
# Spec Review: [spec 文件名]

**审查时间：** [时间戳]
**审查来源：** [对话上下文 / idea-refine 文档 / spec 目标反推]

## 总体评估

| 层级          | 检查项数 | 通过 | 问题 |
| ------------- | -------- | ---- | ---- |
| L1 结构完整性 | —        | —    | —    |
| L2 需求一致性 | —        | —    | —    |
| L3 边界鲁棒性 | —        | —    | —    |

**结论：** ✓ 可以开始计划 / ⚠ 有建议但可继续 / ✗ 建议修复后再计划

## L1 结构完整性

| #   | 检查项       | 结果 | 说明       |
| --- | ------------ | ---- | ---------- |
| 1   | 7 个固定章节 | ✓    | 全部存在   |
| 2   | 方案设计结构 | ⚠    | 缺少架构图 |
| —   | —            | —    | —          |

## L2 需求一致性

| #   | 需求项 | 覆盖情况 | spec 章节 | 说明 |
| --- | ------ | -------- | --------- | ---- |
| —   | —      | —        | —         | —    |

## L3 边界与鲁棒性

| #   | 场景 | 结果 | 说明 |
| --- | ---- | ---- | ---- |
| —   | —    | —    | —    |

## 修复建议

1. [严重/重要/建议] [问题] → [具体修复建议]
```

### 步骤 6：呈现报告 → 交接

将完整的审查报告呈现给用户。用以下格式总结关键发现：

> "审查完成。L1 通过 X/Y，L2 通过 X/Y，L3 通过 X/Y。发现 N 个严重问题、M 个重要建议、K 个可选建议。总体建议：[可以开始 / 有建议但可继续 / 建议修复后再计划]。"

<硬门控>
报告呈现后，根据审查结论决定下一步——**严禁在任何审查结论下直接跳到编写代码**：

- **✓ 可以开始计划** → 告知用户执行命令`/plan`或调用 `planning-and-task-breakdown` 技能创建详细实现计划
- **⚠ 有建议但可继续** → 列出建议项，让用户决定是先修复还是边做边修，确认后执行命令`/plan`
- **✗ 建议修复后再计划** → 列出必须修复的严重问题，待用户修复 spec 后重新审查（不进入 `/plan`）

无论审查结论如何，规范审查之后的下一步是**计划**而非**代码**。跳过 `/plan` 直接实现等同于拿着未经检视的蓝图开始施工。
</硬门控>

## 独立子代理派发说明

执行审查时，使用以下方式派发独立子代理：

```markdown
使用 `Task` 工具分派规范审查子代理。
prompt 中必须包含：

1. 审查任务说明（引用本技能的三层校验标准）
2. Spec 文档的完整路径
3. 原始需求上下文（对话中用户表达的需求、澄清结论）
4. 关联的 idea-refine 文档路径（如存在）
```

子代理应只读——仅执行 Read/Glob/Grep 操作，不得修改任何文件。

## 常见借口

| 借口                       | 现实                                                                                            |
| -------------------------- | ----------------------------------------------------------------------------------------------- |
| "spec 已经自查过了"        | 自查是同一个人检查自己的作业。独立审查才能发现作者看不见的盲点。                                |
| "审查太花时间"             | 修复下游问题的成本是指数级的。5 分钟审查可能节省数小时返工。                                    |
| "用户已经批准了 spec"      | 用户批准的是需求方向，不是文档质量。格式误差和遗漏用户不会注意到，但 planner 会因此跑偏。       |
| "这个 spec 太简单了"       | 简单的 spec 仍然可能遗漏边界条件。审查会根据 spec 复杂度调整深度——简单 spec 审查只需 1-2 分钟。 |
| "planner 可以发现这些问题" | planner 的工作是拆解任务，不是审查规范。让 planner 边审查边规划，两者都做不好。                 |

## 危险信号

- 在当前会话中直接执行审查，而不是派发独立子代理
- 审查报告全为 ✓，没有发现任何问题（可能是敷衍审查）
- 审查了不存在或不完整的 spec 文件
- L3 边界检查时只关注技术边界，忽略用户体验边界（空状态、加载状态）
- 对大型 spec 整体审查而非按子系统分批，导致报告过长无人阅读

## 验证

完成审查后，请确认：

- [ ] 审查由独立子代理执行（非当前会话直接执行）
- [ ] Spec 文档已定位并完整读取
- [ ] 原始需求基线已建立（标注了置信度来源）
- [ ] 三层校验（L1/L2/L3）全部完成
- [ ] 审查报告包含具体章节位置和修复建议
- [ ] 报告已呈现给用户，等待用户决定是否执行命令`/plan` 或 调用 `planning-and-task-breakdown` 技能
