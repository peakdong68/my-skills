# 发布与上线

## 概述

有信心地发布。目标不仅仅是部署——而是安全地部署，同时确保监控就位、回滚方案准备就绪，并清晰了解成功的定义。每一次发布都应该是可逆、可观测且渐进的。

## 适用场景

- 首次将功能部署到生产环境
- 向用户发布重大变更
- 迁移数据或基础设施
- 开放测试版或早期访问计划
- 任何存在风险（所有部署都有风险）的部署

## 预发布检查清单

### 代码质量

- [ ] 所有测试通过（单元、集成、端到端）
- [ ] 构建成功，无警告
- [ ] 代码检查与类型检查通过
- [ ] 代码已审查并批准
- [ ] 没有应在发布前解决的 TODO 注释和占位符
- [ ] 生产代码中没有 `console.log` 调试语句
- [ ] 错误处理覆盖预期的故障模式

### 安全性

- [ ] 代码或版本控制中没有密钥
- [ ] `npm audit` 未显示严重或高危漏洞
- [ ] 对所有面向用户的端点进行输入验证
- [ ] 认证与授权检查到位
- [ ] 安全标头已配置（CSP、HSTS 等）
- [ ] 对认证端点实施速率限制
- [ ] CORS 配置为特定源（非通配符）

### 性能

- [ ] Core Web Vitals 指标在"良好"阈值内
- [ ] 关键路径中没有 N+1 查询
- [ ] 图片已优化（压缩、响应式尺寸、懒加载）
- [ ] 包体积在预算范围内
- [ ] 数据库查询有适当的索引
- [ ] 为静态资源和重复查询配置了缓存

### 可访问性

- [ ] 所有可交互元素都支持键盘导航
- [ ] 屏幕阅读器能传达页面内容和结构
- [ ] 颜色对比度符合 WCAG 2.1 AA 标准（文本对比度 4.5:1）
- [ ] 模态框和动态内容的焦点管理正确
- [ ] 错误消息描述清晰，并与表单字段关联
- [ ] axe-core 或 Lighthouse 中没有可访问性警告

### 基础设施

- [ ] 生产环境变量已设置
- [ ] 数据库迁移已应用（或准备应用）
- [ ] DNS 和 SSL 已配置
- [ ] 静态资源 CDN 已配置
- [ ] 日志和错误报告已配置
- [ ] 健康检查端点存在并能响应

### 文档

- [ ] README 已更新，包含任何新的配置要求
- [ ] API 文档是最新的
- [ ] 已为所有架构决策编写 ADR
- [ ] Changelog 已更新
- [ ] 面向用户的文档已更新（如适用）

## 功能标志策略

利用功能标志发布，将部署与发布解耦：

```typescript
// 功能标志检查
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 新功能：任务共享
  return <TaskSharingPanel task={task} />;
}

// 默认：现有行为
return null;
```

**功能标志生命周期：**

```
1. 部署并确保标志关闭   → 代码已在生产环境中，但未激活
2. 为团队/测试版启用    → 在生产环境中进行内部测试
3. 逐步推广             → 5% → 25% → 50% → 100% 用户
4. 在每个阶段监控        → 观察错误率、性能、用户反馈
5. 清理                 → 完全推广后移除标志和死代码路径
```

**规则：**

- 每个功能标志都有负责人和过期日期
- 在完全推广后 2 周内清理标志
- 不要嵌套功能标志（会造成指数级组合）
- 在 CI 中测试两种标志状态（开启和关闭）

## 分阶段发布

### 发布流程

```
1. 部署到预发布环境(Staging)
   └── 在预发布环境运行完整测试套件
   └── 手动冒烟测试关键流程

2. 部署到生产环境（功能标志关闭）
   └── 验证部署成功（健康检查）
   └── 检查错误监控（无新错误）

3. 为团队启用（为内部用户开启标志）
   └── 团队在生产环境中使用该功能
   └── 24 小时监控窗口

4. 金丝雀发布（为 5% 用户开启标志）
   └── 监控错误率、延迟、用户行为
   └── 对比指标：金丝雀组 vs. 基线组
   └── 24-48 小时监控窗口
   └── 仅在通过所有阈值后继续（见下表）

5. 逐步增加（25% -> 50% -> 100%）
   └── 每一步都进行相同的监控
   └── 随时可以回滚到上一个百分比

6. 完全推广（为所有用户开启标志）
   └── 监控 1 周
   └── 清理功能标志
```

### 发布决策阈值

在每个阶段，使用这些阈值决定是推进、暂停观察还是回滚：

| 指标           | 推进（绿）        | 暂停并调查（黄）            | 回滚（红）                  |
| -------------- | ----------------- | --------------------------- | --------------------------- |
| 错误率         | 在基线的 10% 以内 | 超出基线 10-100%            | 超过基线 2 倍               |
| P95 延迟       | 在基线的 20% 以内 | 超出基线 20-50%             | 超出基线 50% 以上           |
| 客户端 JS 错误 | 无新的错误类型    | 新错误出现，影响 <0.1% 会话 | 新错误出现，影响 >0.1% 会话 |
| 业务指标       | 中性或积极        | 下降 <5%（可能是噪声）      | 下降 >5%                    |

### 何时回滚

出现以下情况立即回滚：

- 错误率增幅超过基线 2 倍
- P95 延迟增幅超过 50%
- 用户报告问题激增
- 检测到数据完整性问题
- 发现安全漏洞

## 监控与可观测性

### 需要监控什么

```
应用程序指标：
├── 错误率（总计和按端点）
├── 响应时间（p50、p95、p99）
├── 请求量
├── 活跃用户
└── 关键业务指标（转化率、参与度）

基础设施指标：
├── CPU 和内存利用率
├── 数据库连接池使用率
├── 磁盘空间
├── 网络延迟
└── 队列深度（如适用）

客户端指标：
├── Core Web Vitals（LCP、INP、CLS）
├── JavaScript 错误
├── 客户端视角的 API 错误率
└── 页面加载时间
```

### 错误报告

```typescript
// 设置带有报告功能的错误边界
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 向错误追踪服务报告
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// 服务端错误报告
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // 不要向用户暴露内部信息
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' },
  });
});
```

### 上线后验证

上线后第一个小时内：

```
1. 检查健康端点返回 200
2. 检查错误监控仪表板（无新错误类型）
3. 检查延迟仪表板（无性能退化）
4. 手动测试关键用户流程
5. 验证日志正在流动且可读
6. 确认回滚机制有效（如可能，进行演习）
```

## 回滚策略

每次部署前都需要制定回滚方案：

```markdown
## [功能/发布] 的回滚方案

### 触发条件

- 错误率 > 基线 2 倍
- P95 延迟 > [X]ms
- 用户报告 [具体问题]

### 回滚步骤

1. 禁用功能标志（如适用）
   或
2. 部署上一个版本：`git revert <commit> && git push`
3. 验证回滚：健康检查、错误监控
4. 沟通：通知团队已回滚

### 数据库注意事项

- 迁移 [X] 有对应的回滚操作：`npx prisma migrate rollback`
- 新功能插入的数据：[保留 / 清理]

### 回滚耗时

- 功能标志：< 1 分钟
- 重新部署上一个版本：< 5 分钟
- 数据库回滚：< 15 分钟
```

## 常见借口

| 借口                                   | 事实                                                         |
| -------------------------------------- | ------------------------------------------------------------ |
| "在预发布环境没问题，生产环境也没问题" | 生产环境的数据、流量模式和边缘情况都不同。部署后需要监控。   |
| "我们不需要为此使用功能标志"           | 每个功能都能受益于终止开关。即使"简单"的变更也可能导致问题。 |
| "监控是额外负担"                       | 没有监控意味着你只能从用户投诉中发现问题，而不是通过仪表板。 |
| "我们稍后再添加监控"                   | 上线前就添加。你无法观察你看不到的东西。                     |
| "回滚等于承认失败"                     | 回滚是负责任的工程实践。发布有问题的功能才是失败。           |

## 危险信号

- 部署时没有回滚方案
- 生产环境没有监控或错误报告
- "大爆炸"式发布（一次性发布所有内容，无分阶段）
- 功能标志没有过期时间或负责人
- 部署后第一个小时内无人监控
- 生产环境配置靠记忆而非代码
- "现在是周五下午，我们发布吧"

## 验证

部署前：

- [ ] 预发布检查清单已完成（所有部分都是绿色）
- [ ] 功能标志已配置（如适用）
- [ ] 回滚方案已记录
- [ ] 监控仪表板已设置
- [ ] 已通知团队部署事宜

部署后：

- [ ] 健康检查返回 200
- [ ] 错误率正常
- [ ] 延迟正常
- [ ] 关键用户流程正常工作
- [ ] 日志正在流动
- [ ] 回滚已测试或已确认就绪

## 发布后归档

当 `/ship` 产出 **发布 (GO)** 决策后，提示用户执行 `/archive` 命令将已完成的功能目录归档：

```
发布决策为 GO。功能已完成并通过所有发布前检查。
  如需归档该功能的 spec 目录，请执行:
    /archive docs/spec/feature_<date>_<id>_<topic>/
```

归档应在 `/ship` 通过之后执行——如果发布检查发现阻塞项需要返工，功能目录必须保留在 `docs/spec/` 活跃区。
